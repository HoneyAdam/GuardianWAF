## Summary

No mass assignment vulnerabilities were found. The codebase uses safe patterns throughout:
- **Dashboard API**: Decodes into typed structs with explicit field allowlists, not raw domain models
- **Tenant API**: Uses DTOs (`CreateTenantRequest`, `UpdateTenantRequest`) with controlled field sets

**Risk:** None. No user input is bound directly to domain models with protected fields.

### 2. Tenant API (`internal/tenant/handlers.go`)

Uses two DTOs:

```go
type CreateTenantRequest struct {
    Name        string        `json:"name"`
    Description string        `json:"description"`
    Domains     []string      `json:"domains"`
    Quota       *ResourceQuota `json:"quota,omitempty"`
}

type UpdateTenantRequest struct {
    Name        string           `json:"name,omitempty"`
    Description string           `json:"description,omitempty"`
    Active      *bool            `json:"active,omitempty"`
    Domains     []string         `json:"domains,omitempty"`
    Quota       *ResourceQuota   `json:"quota,omitempty"`
    Config      *config.Config   `json:"config,omitempty"` // Full tenant config
}
```

`CreateTenantRequest` has a controlled field set. `UpdateTenantRequest` includes `Config *config.Config` which allows setting the full WAF config — this is intentional for per-tenant configuration management but requires the admin API key (`X-API-Key` header check at line 31).

`PublicTenant` (line 135) intentionally excludes sensitive fields (`APIKeyHash`) from API responses.

**Risk:** Low. The `Config` field accepting `*config.Config` is a design choice for tenant config management, not a mass assignment of protected fields like roles or admin flags. All endpoints require API key authentication.
